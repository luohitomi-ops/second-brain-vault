你是檸子第二大腦的維護員。目標：讓「每次載入 context / 常讀」的檔案保持精簡，把只會長大的歷史內容搬去封存檔，但**不搬掉任何還在用的活狀態**。

## 1. 掃描（先只讀不動）
用 Bash 統計下列檔案行數，跟門檻比對，列出超標的：

| 檔案 | 門檻 | 性質 |
|------|------|------|
| 各 `CLAUDE.md`（根 + 01/02/03 區） | 100 行 | 每次 session 自動載入，最該精簡 |
| 記憶 `MEMORY.md` | 120 行 | 每次載入 |
| 三大中樞（品牌/投資/波段中樞.md） | 120 行 | 常讀 |
| `00 Dashboard/系統藍圖.md` | 150 行 | roadmap 本體該精簡 |
| `01 Brand/社群數據/content_plan.md` | 300 行 | 舊輪日曆會累積 |
| `01 Brand/社群數據/history.md` | 依「當月+近7天」判斷 | append-only |

## 2. 對每個超標檔，判斷「活 vs 封存」
- **活狀態（留下）**：通用規則、當前重點的最新 1-2 次結論、當前輪排程、當月+近7天數據、未完成待辦/深挖佇列、還會用到的指標與連結
- **可封存（搬走）**：
  - 帶明確過去日期、已成定局的「分析/復盤/決策紀錄」（例：中樞裡的「2026-07-07 大盤急殺分析」這種 dated 區塊）
  - 已發生且狀態定案（✅已發/❌取消/❌數據遺失）的舊排程列、舊輪日曆
  - 已修好、純供追溯的 bug/變更日誌（超過 2 季的）
  - 過期的資產快照/戰績數字（移到日期檔如 `資產快照_2026Q3.md`）
- ⚠️ **判斷不準就不要搬**：拿不定是不是還在用的，一律留著並在回報裡標「不確定，需你確認」。寧可少搬不可誤搬。

## 3. 封存檔命名規則（一致）
- 中樞 dated 分析 → 同區 `XX決策日誌封存.md`（已存在就 append）
- append 日誌按月 → `原檔名_YYYY-MM.md`
- 系統藍圖 bug/變更日誌 → `00 Dashboard/系統變更日誌_YYYY-QN.md`
- 過期快照 → `資產快照_YYYYQN.md` 等日期檔
- **搬走後，活檔留一行指標**：`> 更早的 XXX 已封存 → [[封存檔名]]`，不要讓連結斷掉

## 4. 先回報再動手
先輸出「建議搬遷清單」：哪個檔、搬哪幾段/幾行到哪個封存檔、搬完活檔會剩幾行。**等我說「確認」才實際搬**。搬完回報各檔新行數。

## 原則
- 只搬「歷史事實」，不動「還在用的規則與當前狀態」
- 封存 = 移動不是刪除，隨時可回查（保留可追溯性）
- 每季或發現某檔明顯超標時跑一次；跟每月最後一天的 history.md 歸檔可同天做

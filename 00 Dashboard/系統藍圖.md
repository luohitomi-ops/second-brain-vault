## 設計哲學（參考 Houch 21天 + Wenhao panopticon）

1. **單一底層**：所有東西都是 `C:\tia\vault` 裡的 Markdown，不分散。
2. **節點化**：每個工作流 / 專案 / 顧問群 / 決策 = 一個節點（一個 .md），用 `[[連結]]` 互相串成神經網路。
3. **AI 操作大腦**：Claude Code 讀寫這些節點，幫你串連、總結、更新——你不用手動整理。
4. **可隨時新增**：新工作流 = 新增一個節點檔，自動進入神經網路。
5. **零 API 成本**：用 Claude Pro + Claude Code，不需另外的 API key。

## 🧹 系統維護節奏（2026-07-01 新增，防 context 膨脹）

> 背景：CLAUDE.md 每次開 session 自動載入 context，會一直長大；history.md 類知識庫檔案雖不自動載入，但單次 Read 量太大也拖 context。定期瘦身維持效能。
> **核心原則（2026-07-13 定調）**：每個會長大的檔案都要「活狀態 vs 歷史日誌」分離——活檔只留當前狀態+近期，歷史事實搬去封存檔（移動不刪除，留指標連結）。統一用 **`/瘦身`** 指令半自動掃描+提建議+確認後搬，不靠人記得。

| 動作 | 頻率 | 內容 |
|------|------|------|
| **`/瘦身` 全庫掃描** | 每季 or 發現某檔明顯超標時 | 掃 CLAUDE.md(<100行)/中樞(<120)/系統藍圖(<150)/content_plan(<300) 等門檻，超標的列「活 vs 封存」建議，確認後搬去封存檔。取代下面各項的手動判斷，是總入口 |
| **history.md 歸檔** | 每月最後一天 | 把上個月的行搬到 `history_YYYY-MM.md`，主檔只留當月+近7天。跟月度收支復盤同天做，養成固定動作 |
| **CLAUDE.md 瘦身** | 每季（併入 `/瘦身`） | 把「某天資產快照/戰績數字」等會過期的內容移到日期檔（如 `資產快照_2026Q3.md`），CLAUDE.md 只留通用規則 |
| 中樞 dated 分析歸檔 | 每季（併入 `/瘦身`） | 中樞「當前重點」只留最新1-2次結論，舊的 dated 分析/復盤搬去 `XX決策日誌封存.md` |
| 系統藍圖 bug/變更日誌 | 每季（併入 `/瘦身`） | roadmap 本體留現況，超過2季的 bug/變更日誌搬 `系統變更日誌_YYYY-QN.md` |
| 波段資訊庫/投資知識庫歸檔 | 每季（量少，之後量變大再改月） | 同上邏輯 |
| **發文靈感池：未使用 Brief 清理** | 每週日 02:05 自動（已上線 2026-07-10） | 話題有熱度時效性，超過 7 天且「已使用」仍是 ⬜ 的 `01 Brand/Brief *.md` 自動刪除，已使用（✅）的不受影響；手動可打 `/清理Brief [天數]` |

## ⚠️ 2026-07-12 已知問題：Brief 生成偶爾撞 Google 搜尋 grounding 免費額度 429（暫緩處理）
- 現象：`gemini-brief.mjs`（Daily Brief）2026-07-12 凌晨02:00自動排程失敗，Gemini 429，5次重試（已加強重試機制+讀取Google建議等待秒數）全部失敗
- **根因排查**：實測發現不是一般文字生成額度問題（純文字請求測試回200正常），是 **Google 搜尋 grounding 功能的免費額度**（Brief 生成必須用 `tools:[{google_search:{}}]` 才能強制要求真實來源URL）——這是跟一般生成分開計算的獨立配額，額度比一般生成小很多，且錯誤沒附「建議等待秒數」，代表是額度見底不是短暫尖峰，同一天內重試無效
- 全庫查過，只有 `gemini-brief.mjs` 用到 google_search grounding，其他 `/整理`／`/財報`／`/研究`／`/建模` 都沒用（不是被其他腳本瓜分掉的）
- **已改善**：`gemini-brief.mjs`（背景排程，5次重試+封頂5分鐘+讀取RetryInfo）、`gemini-organize.mjs`（互動指令，補上原本完全沒有的429重試，封頂45秒）
- **真正解法（使用者決定「之後再說」，暫不處理）**：幫這個 Google Cloud 專案開啟計費解除免費額度天花板（Brief一天用量極小，實際費用應該接近零，但需要使用者自己去 Google Cloud Console 綁信用卡，Claude 不會也不該代為操作付款設定）；不打算拿掉 grounding 換取不撞429，因為那樣就失去「強制真實來源URL」的設計初衷
- 通常額度會在隔天（美西午夜，約台灣下午3-4點）重置，多數情況下次日02:00排程會自動恢復正常

**⚠️ 2026-07-15 更新：實際不是「隔天恢復」，是連續卡死**——查證 `01 Brand/Brief*.md` 檔案清單，最新一份只到 7/10，7/11~7/15 整整 5 天完全沒有成功生成過，代表這個免費 grounding 額度見底後**不是每日重置**，可能是週期更長（甚至月）或這個專案的免費配額本來就趨近於零。
- **已改善（省資源）**：`gemini-brief.mjs` 的重試邏輯改成分流處理——429 且 Google 沒附「建議等待秒數」= 真的額度見底不是短暫尖峰，**直接快速失敗不再重試5次**（省時間+省API呼叫）；503 或 429 附建議等待秒數的才照原本邏輯重試，這種才是真的值得等的短暫尖峰
